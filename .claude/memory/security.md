# Security Issues & Findings

## RESOLVED: api-keys.json Committed to Git

**Fixed.** `api-keys.json` added to `.gitignore` and removed from git index (`git rm --cached`). The file no longer appears in future commits. Rotate any previously exposed keys.

## RESOLVED: .admin-key Not Gitignored

**Fixed.** `.admin-key` added to `.gitignore`. File stays on disk but won't be committed.

## RESOLVED: Stripe Webhook Signature Bypass

**Fixed.** Webhook endpoint now requires `STRIPE_WEBHOOK_SECRET`. If env var not set, returns `400 WEBHOOK_NOT_CONFIGURED` â€” no unsigned events are processed.

```js
// stripe-billing.mjs â€” current behavior
if (!webhookSecret) {
  return res.status(400).json({ error: 'Webhook not configured', code: 'WEBHOOK_NOT_CONFIGURED' });
}
event = stripe.webhooks.constructEvent(req.rawBody, sig, webhookSecret);
```

## RESOLVED: /billing/signup Had No Rate Limiting

**Fixed.** `signupLimiter` applied: 5 requests per 15 minutes per IP.

## HIGH: Default Admin Key (still active)

```js
const ADMIN_KEY = process.env.ADMIN_KEY || 'admin-secret-key';
```
If `ADMIN_KEY` env var is not set, any request with `X-Admin-Key: admin-secret-key` gains full admin access. The systemd service file has a TODO placeholder. **Always set this before deployment.**

## MEDIUM: CORS Allows All Origins (intentional)

```js
app.use(cors());
```
No origin restrictions. Intentional for a public API. If ever converted to a private internal API, add `origin` config.

## MEDIUM: Email-Derived API Keys (stripe-billing)

Keys generated by `stripe-billing.mjs` are derived from base64 of the customer's email â€” predictable. Anyone who knows a user's email can compute their API key.

```js
`ts_free_${Buffer.from(email).toString('base64').replace(/[+/=]/g, '').slice(0, 24)}`
```

The `/billing/signup` endpoint in `api-server.mjs` uses `uuid` (random) â€” preferred. Keys from Stripe checkout still use the email-derived pattern.

## MEDIUM: JSON File Storage â€” No Concurrency Protection

`api-keys.json`, `usage.json`, `customers.json`, `subscriptions.json` have no locking. Under concurrent load:
- Usage counts can be inaccurate
- Simultaneous key creates could conflict on write
- No atomic write â€” crash mid-write can corrupt the file

Fix: use a proper database (SQLite minimum, Redis/Postgres for production scale).

## LOW: API Key Soft-Delete Only

Revoked keys are set to `active: false` but remain in `api-keys.json`. Old revoked keys accumulate indefinitely.

## LOW: Usage Saves Every 10 Requests

Up to 9 usage records can be lost on crash. Tracked stats will be slightly inaccurate.

## LOW: Unrestricted `tier` on Admin Key Creation

`POST /admin/keys` accepts any string for `tier`. An unknown tier (e.g., `"unlimited"`) creates a key that falls back to `free` rate limits â€” no privilege escalation, but confusing.

## Security Wins (current)

- helmet applied (CSP disabled for image serving, rest active)
- express-rate-limit per API key (not per IP â€” better for multi-tenant)
- Admin endpoints protected by separate `X-Admin-Key` header
- Stripe webhook signature verification enforced
- `/billing/signup` rate-limited by IP (5/15min)
- systemd service uses `NoNewPrivileges`, `PrivateTmp`, `ProtectSystem=strict`
- Raw body capture for Stripe webhook handled before `express.json()` middleware
